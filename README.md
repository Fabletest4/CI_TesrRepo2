# Fraud Detection Alerts Platform

> **Repository status:** Architecture and standards foundation. This repository establishes the target platform design, API contract, and engineering standards for the Fraud Detection Alerts ecosystem. Service implementation is in progress.

---

## Overview

The **Fraud Detection Alerts** platform processes real-time payment transactions, evaluates them for fraud risk, and surfaces actionable alerts to analysts and downstream notification channels. It is built on an event-driven microservice architecture with Apache Kafka at the core.

This repository currently serves as the authoritative reference for:

- Platform architecture and service topology
- Kafka event contracts
- REST API shape and schema
- Backend engineering standards
- Deployment and CI/CD model

---

## Business Value

| Outcome | Description |
|---|---|
| Faster fraud detection | Real-time event processing minimises the window between a suspicious transaction and analyst awareness |
| Consistent risk evaluation | Rule-based scoring engine applies uniform criteria across all payment events |
| Improved analyst efficiency | The admin dashboard surfaces prioritised alerts, reducing manual triage effort |
| Scalable operations | Independent microservices allow ingestion, scoring, and notification to scale separately |
| Auditability | End-to-end event trail from raw transaction to resolved alert supports compliance and investigation workflows |

## Platform Architecture

The platform is composed of five components connected through a Kafka event bus.

```
payment-service ──► [payment-events] ──► fraud-engine ──► [fraud-alerts] ──► notification-service
                                                │                                     │
                                                ▼                              [alert-notifications]
                                        admin-dashboard
```

### Services

| Service | Responsibility |
|---|---|
| **payment-service** | Ingests payment transactions and publishes them as events |
| **fraud-engine** | Consumes payment events, calculates a fraud risk score, and generates alerts for high-risk transactions |
| **notification-service** | Consumes fraud alerts and dispatches email and Slack notifications |
| **admin-dashboard** | Presents fraud alerts to analysts for review and action |

See [`architecture.md`](./architecture.md) for full service descriptions and fraud scoring rules.

---

## Event Flow

1. `payment-service` publishes a payment event to the **`payment-events`** Kafka topic.
2. `fraud-engine` consumes the event and calculates a risk score using documented scoring rules.
3. High-risk transactions (score ≥ 40) produce a fraud alert on the **`fraud-alerts`** topic.
4. `notification-service` consumes the fraud alert and publishes a notification event to **`alert-notifications`**.
5. `admin-dashboard` reads alerts directly and presents them to fraud analysts.

### Kafka Topics

| Topic | Description |
|---|---|
| `payment-events` | Raw payment transactions published by payment-service |
| `fraud-alerts` | Scored fraud alerts generated by the fraud engine |
| `alert-notifications` | Outbound notification events consumed by notification-service |

## Fraud Scoring Model

The fraud engine applies a rule-based scoring model. Each rule contributes to a cumulative risk score:

| Rule | Score Contribution |
|---|---|
| Transaction amount > $10,000 | +40 |
| Foreign IP mismatch | +30 |
| Multiple failed attempts | +25 |
| High-risk country | +50 |

**Severity thresholds:**

| Score | Severity |
|---|---|
| 70 or above | Critical Alert |
| 40 – 69 | Warning Alert |

## API Reference

The platform exposes a JSON REST API under the `/api/v1` prefix. The full contract is defined in [`api-schema.yaml`](./api-schema.yaml) (OpenAPI 3.0).

### Endpoints

#### `GET /api/v1/alerts`
Returns a list of fraud alerts.

#### `GET /api/v1/alerts/{id}`
Returns details for a single fraud alert identified by its `id`.

### `FraudAlert` Schema

| Field | Type | Description |
|---|---|---|
| `id` | `string` | Unique alert identifier |
| `transactionId` | `string` | Reference to the originating transaction |
| `score` | `integer` | Calculated fraud risk score |
| `severity` | `string` | Alert severity (`WARNING` or `CRITICAL`) |
| `createdAt` | `string` (ISO 8601) | Timestamp of alert creation |

> **Note:** All responses are JSON. Correlation IDs, pagination, filtering, and authentication requirements will be reflected in the API schema as the implementation matures.

## Engineering Standards

All backend services follow the conventions documented in [`coding-standards.md`](./coding-standards.md).

### Backend (Java 21)

- **Constructor injection** — all dependencies injected via constructor; field injection is prohibited.
- **Immutable DTOs** — data transfer objects are immutable by design.
- **Structured logging** — log messages include transaction and alert identifiers for traceability.
- **Centralised exception handling** — consistent error responses across all endpoints.
- **Correlation IDs** — propagated through all service boundaries for end-to-end request tracing.
- **No sensitive payment data exposure** — raw card numbers, account numbers, and PII must not appear in logs, API responses, or alert payloads.

### API Design

- All endpoints served under `/api/v1`.
- JSON request and response bodies.
- Consistent error model across services.

### Dashboard (admin-dashboard)

- Accessibility standards apply to all UI components.

### Security

- Sensitive payment data must not be exposed in API responses or logs.
- Input validation is required on all service entry points.
